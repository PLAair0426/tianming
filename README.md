## 📊 限流策略

### 各接口的限流配置

| 接口 | 限流策略 | 说明 |
|------|---------|------|

| `/api/divination/karma-status` | **无限制** | 查询状态，不影响资源 |
| `/api/divination/recharge` | **5次/分钟** | 充能接口，防止滥用 |

## 🔧 工作原理

### 限流算法：滑动窗口

**实现方式**：
- 使用字典存储每个IP的请求时间戳
- 每次请求时清理过期记录（超过时间窗口的）
- 检查当前时间窗口内的请求数
- 如果超过限制，返回429错误

**示例**：
```
时间：00:00:00 - 用户A发送请求1 → 记录
时间：00:00:10 - 用户A发送请求2 → 记录
时间：00:00:20 - 用户A发送请求3 → 记录
...
时间：00:00:50 - 用户A发送请求10 → 记录（达到限制）
时间：00:00:55 - 用户A发送请求11 → ❌ 被限流，返回429错误
时间：00:01:01 - 用户A发送请求12 → ✅ 允许（请求1已过期）
```

---

## 🛡️ 多层保护机制

### 第一层：API 限流
- **作用**：防止单个用户发送过多请求
- **保护**：服务器资源、API调用次数
- **效果**：单个用户最多10次/分钟占卜

### 第二层：元气系统
- **作用**：防止用户频繁使用
- **保护**：通过消耗机制限制使用频率
- **效果**：频繁使用消耗更多元气，元气耗尽无法使用

### 第三层：用户隔离
- **作用**：每个用户独立状态
- **保护**：用户之间互不影响
- **效果**：用户A的消耗不影响用户B

# 多人使用问题分析

## ⚠️ 当前存在的问题

### 1. **全局共享元气系统（严重问题）**

**问题描述**：
- 代码中使用全局单例：`karma_system = KarmaSystem(...)`
- **所有用户共享同一个元气值**
- 用户A消耗元气后，用户B也会受到影响

**影响**：
- 用户A频繁使用，消耗完元气后，所有用户都无法使用
- 用户B可能看到"元气不足"错误，即使他还没用过
- 无法区分不同用户的使用情况

**代码位置**：
```python
# backend/app/routers/divination.py
karma_system = KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0)
```

---

### 2. **Render Free Plan 限制**

**问题描述**：
- 服务在15分钟无活动后会休眠
- 首次请求需要30-60秒冷启动

**影响**：
- 如果服务休眠，所有用户首次访问都需要等待
- 可能被误认为是"用不了"

---

### 3. **没有 API 限流**

**问题描述**：
- 代码中没有实现 API 限流
- 没有并发请求限制

**影响**：
- 如果多人同时使用，可能导致：
  - 资源耗尽（CPU、内存）
  - 响应变慢
  - 服务崩溃
  - DeepSeek API 调用过多，可能触发限流

---

### 4. **没有用户隔离**

**问题描述**：
- 没有用户认证系统
- 没有会话管理
- 无法区分不同用户

**影响**：
- 无法为不同用户提供独立的使用体验
- 无法追踪单个用户的使用情况

---

## ✅ 解决方案

### 方案 1：为每个用户创建独立的元气系统（推荐）

**实现方式**：
- 使用会话ID或IP地址区分用户
- 为每个用户创建独立的 `KarmaSystem` 实例
- 使用字典存储：`{session_id: KarmaSystem}`

**优点**：
- 用户之间互不影响
- 每个用户有独立的元气值
- 实现相对简单

**缺点**：
- 需要会话管理
- 内存占用增加（但通常可接受）

---

### 方案 2：添加 API 限流

**实现方式**：
- 使用 `slowapi` 或 `fastapi-limiter` 库
- 限制每个IP的请求频率
- 例如：每分钟最多10次请求

**优点**：
- 防止滥用
- 保护服务资源
- 防止 DeepSeek API 被限流

**缺点**：
- 需要额外依赖
- 可能影响正常用户（如果限流太严格）

---

### 方案 3：升级 Render 计划

**实现方式**：
- 从 Free Plan 升级到 Starter Plan（$7/月）
- 服务常驻，无冷启动

**优点**：
- 解决休眠问题
- 更好的性能
- 更稳定的服务

**缺点**：
- 需要付费

---

### 方案 4：添加用户认证（长期方案）

**实现方式**：
- 使用 JWT 或 Session 认证
- 为每个用户分配唯一ID
- 在数据库中存储用户状态

**优点**：
- 完整的用户隔离
- 可以追踪使用历史

**缺点**：

# 并发处理优化说明

## ⚠️ 问题：AI调用阻塞其他请求

### 问题描述

**原来的代码**：
```python
# 路由函数是异步的
async def interpret_divination(...):
    # 但AI调用是同步的！
    interpretation = agent.consult(...)  # 同步调用，会阻塞事件循环
```

**问题**：
- 用户A调用AI（需要10-30秒）
- 在这10-30秒内，**事件循环被阻塞**
- 用户B和C的请求**无法处理**，必须等待
- 导致其他用户无法使用

### 影响

```
时间线：
00:00 - 用户A发送占卜请求
00:00 - 开始调用AI（同步，阻塞事件循环）
00:05 - 用户B发送请求 → ❌ 被阻塞，无法处理
00:10 - 用户C发送请求 → ❌ 被阻塞，无法处理
00:15 - AI调用完成
00:15 - 用户B的请求开始处理
00:20 - 用户C的请求开始处理
```

**结果**：用户B和C需要等待用户A的AI调用完成！

---

## ✅ 解决方案：使用异步执行

### 修改后的代码

```python
# 使用 asyncio.to_thread() 将同步调用放到线程池中执行
interpretation = await asyncio.to_thread(
    agent.consult,
    hexagram_data,
    request.question,
    stream_mode=False
)
```

### 工作原理

**`asyncio.to_thread()` 的作用**：
- 将同步函数放到线程池中执行
- **不阻塞事件循环**
- 其他请求可以正常处理

**修改后的时间线**：
```
时间线：
00:00 - 用户A发送占卜请求
00:00 - 开始调用AI（在线程池中执行，不阻塞）
00:05 - 用户B发送请求 → ✅ 立即处理，不等待
00:10 - 用户C发送请求 → ✅ 立即处理，不等待
00:15 - 用户A的AI调用完成
00:15 - 用户B的请求已完成
00:10 - 用户C的请求已完成
```

**结果**：用户B和C可以立即得到响应，不需要等待！

## 🔍 技术细节

### asyncio.to_thread() 原理

```python
# 伪代码说明
async def to_thread(func, *args):
    # 1. 将函数放到线程池中执行
    # 2. 不阻塞事件循环
    # 3. 等待线程执行完成
    # 4. 返回结果
    return await executor.submit(func, *args)
```

### 为什么需要这个？

- FastAPI 使用异步事件循环
- 同步函数会阻塞事件循环
- 使用线程池可以避免阻塞
- 其他请求可以继续处理

# 用户隔离原理说明

## 🔍 当前问题：全局共享状态

### 问题原理

**当前代码结构**：
```python

@router.post("/divination/interpret")

    # 所有用户都使用同一个 karma_system
    karma_system.update_vitality()  # 用户A修改，影响所有用户
    preview = karma_system.calculate_cost_preview()  # 用户B看到的是用户A修改后的状态
    karma_system.commit_transaction(cost)  # 用户C消耗，影响所有用户
```

**内存中的状态**：
```
┌─────────────────────────────────┐
│  全局内存空间                    │
│                                 │
│  karma_system (单例)            │
│  ├─ current_vitality: 50.0     │ ← 所有用户共享这个值
│  ├─ last_cast_time: 1234567890 │
│  ├─ total_uses: 10             │
│  └─ ...                        │
└─────────────────────────────────┘
     ↑              ↑              ↑
  用户A           用户B           用户C
  (共享)          (共享)          (共享)
```

**问题场景**：
1. 用户A访问 → `karma_system.current_vitality = 100`
2. 用户A算卦 → 消耗20，`current_vitality = 80`
3. 用户B访问 → 看到的是80（不是100！）
4. 用户B算卦 → 消耗20，`current_vitality = 60`
5. 用户A再算 → 看到的是60（不是80！）

**结果**：所有用户的状态互相影响，无法独立使用。

## ✅ 解决方案：基于IP的用户隔离

### 核心原理

**使用字典存储每个用户的独立状态**：
```python
from collections import defaultdict

# 字典：{用户标识: KarmaSystem实例}
user_karma_systems = defaultdict(
    lambda: KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0)
)
```

**内存中的状态**：
```
┌─────────────────────────────────────────────┐
│  全局内存空间                                │
│                                             │
│  user_karma_systems (字典)                  │
│  ├─ "192.168.1.100" → KarmaSystem实例A     │
│  │   ├─ current_vitality: 80.0             │ ← 用户A独立
│  │   ├─ last_cast_time: 1234567890        │
│  │   └─ total_uses: 5                     │
│  │                                         │
│  ├─ "192.168.1.101" → KarmaSystem实例B     │
│  │   ├─ current_vitality: 100.0           │ ← 用户B独立
│  │   ├─ last_cast_time: 0                │
│  │   └─ total_uses: 0                     │
│  │                                         │
│  └─ "192.168.1.102" → KarmaSystem实例C     │
│      ├─ current_vitality: 60.0            │ ← 用户C独立
│      ├─ last_cast_time: 1234567891        │
│      └─ total_uses: 3                     │
└─────────────────────────────────────────────┘
     ↑              ↑              ↑
  用户A           用户B           用户C
  (独立)          (独立)          (独立)
```

## 🔧 实现原理详解

### 1. defaultdict 的工作原理

**defaultdict 是什么？**
- Python 标准库中的字典类型
- 当访问不存在的键时，自动创建默认值
- 使用工厂函数生成默认值

**代码示例**：
```python
from collections import defaultdict

# 普通字典：访问不存在的键会报错
normal_dict = {}
# normal_dict["new_key"]  # KeyError!

# defaultdict：自动创建默认值
user_systems = defaultdict(lambda: KarmaSystem(...))
user_systems["new_ip"]  # 自动创建新的 KarmaSystem 实例！
```

**工作流程**：
```
1. 用户A (IP: 192.168.1.100) 首次访问
   → user_karma_systems["192.168.1.100"] 不存在
   → defaultdict 自动调用 lambda 函数
   → 创建新的 KarmaSystem(max_vitality=100.0, ...)
   → 存储到字典中

2. 用户A 再次访问
   → user_karma_systems["192.168.1.100"] 已存在
   → 直接返回已创建的实例
   → 保持状态（元气值、使用次数等）

3. 用户B (IP: 192.168.1.101) 首次访问
   → user_karma_systems["192.168.1.101"] 不存在
   → 创建新的独立实例
   → 与用户A完全隔离
```

**修改后的代码流程**：


**流程图**：

```
    ↓

```

### 3. 状态隔离原理

**关键点**：每个用户有独立的实例，状态互不影响

**示例场景**：

```
时间线：
T1: 用户A (IP: 1.1.1.1) 访问
    → user_karma_systems["1.1.1.1"] = KarmaSystem(100%)
    → 算卦，消耗20%
    → 状态：80%

T2: 用户B (IP: 2.2.2.2) 访问
    → user_karma_systems["2.2.2.2"] = KarmaSystem(100%)  ← 新实例！
    → 算卦，消耗20%
    → 状态：80%（但这是用户B的，不影响用户A）

T3: 用户A 再次访问
    → user_karma_systems["1.1.1.1"]  ← 使用已有实例
    → 看到状态：80%（用户A自己的状态）
    → 算卦，消耗20%
    → 状态：60%

T4: 用户B 再次访问
    → user_karma_systems["2.2.2.2"]  ← 使用已有实例
    → 看到状态：80%（用户B自己的状态，不受用户A影响）
```

## 🔄 生命周期管理

### 实例创建

**自动创建**：
- 新用户首次访问时，defaultdict 自动创建
- 无需手动管理

**创建时机**：
```python
# 用户首次访问时

# ↑ 自动调用 lambda: KarmaSystem(...)
# ↑ 创建新实例并存储
```

### 实例保持

**持久化**：
- 实例存储在内存中（字典中）
- 只要服务不重启，状态会一直保持
- 用户再次访问时，使用同一个实例

**状态保持**：
```python
# 用户A第一次访问
karma_system_A = user_karma_systems["1.1.1.1"]
karma_system_A.current_vitality = 80.0

# 用户A第二次访问（可能是几小时后）
karma_system_A = user_karma_systems["1.1.1.1"]  # 同一个实例
print(karma_system_A.current_vitality)  # 仍然是 80.0（如果没恢复）
```

### 实例清理（可选）

**问题**：如果用户很多，内存会一直增长

**解决方案**：定期清理不活跃的用户
```python
import time

def cleanup_inactive_users(max_idle_time=3600):  # 1小时
    """清理1小时未活跃的用户"""
    current_time = time.time()
    inactive_ips = []
    
    for ip, system in user_karma_systems.items():
        idle_time = current_time - system.last_active_time
        if idle_time > max_idle_time:
            inactive_ips.append(ip)
    
    for ip in inactive_ips:
        del user_karma_systems[ip]
    
    return len(inactive_ips)
```

## ⚖️ 优缺点分析

### 优点

1. **简单易实现**
   - 只需修改少量代码
   - 使用标准库，无需额外依赖

2. **自动管理**
   - defaultdict 自动创建实例
   - 无需手动管理生命周期

3. **状态隔离**
   - 每个用户独立状态
   - 互不影响

4. **性能良好**
   - 字典查找 O(1) 时间复杂度
   - 内存占用小

### 缺点

1. **IP 地址限制**
   - 同一网络下的用户共享IP（NAT）
   - 这些用户会共享元气系统

2. **IP 可能变化**
   - 用户切换网络，IP变化
   - 会创建新实例，丢失之前的状态

3. **内存增长**
   - 用户越多，内存占用越大
   - 需要定期清理（可选）

4. **不是真正的用户识别**
   - 基于IP，不是基于用户账号
   - 无法跨设备保持状态

## 🎯 原理总结

### 核心思想

**从"全局单例"改为"字典映射"**：

```
修改前：1个实例 → 所有用户共享
修改后：N个实例 → 每个用户独立
```

### 关键技术

1. **defaultdict**：自动创建默认值
2. **字典映射**：IP → KarmaSystem实例
3. **请求对象**：从 Request 获取用户IP

### 工作流程

```
请求 → 获取IP → 查找/创建实例 → 使用实例 → 返回结果
```

### 在 divination.py 中添加限流

```python
from collections import defaultdict
from time import time

# 存储每个IP的请求记录（用于限流）
request_records = defaultdict(list)

def check_rate_limit(
    client_ip: str, 
    max_requests: int = 10, 
    window_seconds: int = 60
) -> tuple[bool, dict]:
    """
    检查是否超过限流
    :param client_ip: 客户端IP
    :param max_requests: 时间窗口内最大请求数
    :param window_seconds: 时间窗口（秒）
    :return: (是否允许, 限流信息)
    """
    current_time = time()
    
    # 清理过期记录
    request_records[client_ip] = [
        req_time for req_time in request_records[client_ip]
        if current_time - req_time < window_seconds
    ]
    
    # 检查是否超过限制
    if len(request_records[client_ip]) >= max_requests:
        # 计算需要等待的时间
        oldest_request = min(request_records[client_ip])
        retry_after = int(window_seconds - (current_time - oldest_request))
        
        return False, {
            "max_requests": max_requests,
            "window_seconds": window_seconds,
            "current_requests": len(request_records[client_ip]),
            "retry_after": retry_after
        }
    
    # 记录本次请求
    request_records[client_ip].append(current_time)
    return True, {}

# 在路由中使用
@router.post("/divination/interpret")
async def interpret_divination(request: DivinationRequest, req: Request):
    client_ip = get_client_ip(req)
    
    # 检查限流（每分钟最多10次）
    allowed, limit_info = check_rate_limit(client_ip, max_requests=10, window_seconds=60)
    if not allowed:
        raise HTTPException(
            status_code=429,
            detail={
                "error": "请求过于频繁",
                "message": f"请稍后再试，每分钟最多10次请求。请在 {limit_info['retry_after']} 秒后重试。",
                "limit_info": limit_info
            }
        )
    
    # 后续逻辑...
```
