## 🔧 工作原理

### 限流算法：滑动窗口

**实现方式**：
- 使用字典存储每个IP的请求时间戳
- 每次请求时清理过期记录（超过时间窗口的）
- 检查当前时间窗口内的请求数
- 如果超过限制，返回429错误

**示例**：
```
时间：00:00:00 - 用户A发送请求1 → 记录
时间：00:00:10 - 用户A发送请求2 → 记录
时间：00:00:20 - 用户A发送请求3 → 记录
...
时间：00:00:50 - 用户A发送请求10 → 记录（达到限制）
时间：00:00:55 - 用户A发送请求11 → ❌ 被限流，返回429错误
时间：00:01:01 - 用户A发送请求12 → ✅ 允许（请求1已过期）
```

## 🛡️ 多层保护机制

### 第一层：API 限流
- **作用**：防止单个用户发送过多请求
- **保护**：服务器资源、API调用次数
- **效果**：单个用户最多10次/分钟占卜

### 第二层：元气系统
- **作用**：防止用户频繁使用
- **保护**：通过消耗机制限制使用频率
- **效果**：频繁使用消耗更多元气，元气耗尽无法使用

### 第三层：用户隔离
- **作用**：每个用户独立状态
- **保护**：用户之间互不影响
- **效果**：用户A的消耗不影响用户B

## 🎯 效果预期

### 场景 1：正常使用
- 用户正常使用，不会触发限流
- 每分钟10次占卜已经足够

### 场景 2：频繁使用
- 用户快速连续点击
- 超过10次/分钟后，返回429错误
- 提示用户等待，不会影响其他用户

### 场景 3：多用户同时使用
- 用户A频繁使用 → 被限流，但不影响用户B
- 用户B正常使用 → 不受影响
- 服务资源得到保护

## 📊 限流参数建议

### 根据用户量调整

**用户量 < 10人**：
- 占卜解读：10次/分钟（当前设置）
- 生成爻：30次/分钟（当前设置）

**用户量 10-50人**：
- 占卜解读：8次/分钟（更严格）
- 生成爻：20次/分钟

**用户量 > 50人**：
- 占卜解读：5次/分钟（更严格）
- 生成爻：15次/分钟
- 考虑升级服务器
