# 用户隔离原理说明

## 🔍 当前问题：全局共享状态

### 问题原理

**当前代码结构**：
```python
# backend/app/routers/divination.py

# 全局单例 - 所有用户共享这一个实例
karma_system = KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0)

@router.post("/divination/interpret")
async def interpret_divination(request: DivinationRequest):
    # 所有用户都使用同一个 karma_system
    karma_system.update_vitality()  # 用户A修改，影响所有用户
    preview = karma_system.calculate_cost_preview()  # 用户B看到的是用户A修改后的状态
    karma_system.commit_transaction(cost)  # 用户C消耗，影响所有用户
```

**内存中的状态**：
```
┌─────────────────────────────────┐
│  全局内存空间                    │
│                                 │
│  karma_system (单例)            │
│  ├─ current_vitality: 50.0     │ ← 所有用户共享这个值
│  ├─ last_cast_time: 1234567890 │
│  ├─ total_uses: 10             │
│  └─ ...                        │
└─────────────────────────────────┘
     ↑              ↑              ↑
  用户A           用户B           用户C
  (共享)          (共享)          (共享)
```

**问题场景**：
1. 用户A访问 → `karma_system.current_vitality = 100`
2. 用户A算卦 → 消耗20，`current_vitality = 80`
3. 用户B访问 → 看到的是80（不是100！）
4. 用户B算卦 → 消耗20，`current_vitality = 60`
5. 用户A再算 → 看到的是60（不是80！）

**结果**：所有用户的状态互相影响，无法独立使用。

---

## ✅ 解决方案：基于IP的用户隔离

### 核心原理

**使用字典存储每个用户的独立状态**：
```python
from collections import defaultdict

# 字典：{用户标识: KarmaSystem实例}
user_karma_systems = defaultdict(
    lambda: KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0)
)
```

**内存中的状态**：
```
┌─────────────────────────────────────────────┐
│  全局内存空间                                │
│                                             │
│  user_karma_systems (字典)                  │
│  ├─ "192.168.1.100" → KarmaSystem实例A     │
│  │   ├─ current_vitality: 80.0             │ ← 用户A独立
│  │   ├─ last_cast_time: 1234567890        │
│  │   └─ total_uses: 5                     │
│  │                                         │
│  ├─ "192.168.1.101" → KarmaSystem实例B     │
│  │   ├─ current_vitality: 100.0           │ ← 用户B独立
│  │   ├─ last_cast_time: 0                │
│  │   └─ total_uses: 0                     │
│  │                                         │
│  └─ "192.168.1.102" → KarmaSystem实例C     │
│      ├─ current_vitality: 60.0            │ ← 用户C独立
│      ├─ last_cast_time: 1234567891        │
│      └─ total_uses: 3                     │
└─────────────────────────────────────────────┘
     ↑              ↑              ↑
  用户A           用户B           用户C
  (独立)          (独立)          (独立)
```

## 🔧 实现原理详解

### 1. defaultdict 的工作原理

**defaultdict 是什么？**
- Python 标准库中的字典类型
- 当访问不存在的键时，自动创建默认值
- 使用工厂函数生成默认值

**代码示例**：
```python
from collections import defaultdict

# 普通字典：访问不存在的键会报错
normal_dict = {}
# normal_dict["new_key"]  # KeyError!

# defaultdict：自动创建默认值
user_systems = defaultdict(lambda: KarmaSystem(...))
user_systems["new_ip"]  # 自动创建新的 KarmaSystem 实例！
```

**工作流程**：
```
1. 用户A (IP: 192.168.1.100) 首次访问
   → user_karma_systems["192.168.1.100"] 不存在
   → defaultdict 自动调用 lambda 函数
   → 创建新的 KarmaSystem(max_vitality=100.0, ...)
   → 存储到字典中

2. 用户A 再次访问
   → user_karma_systems["192.168.1.100"] 已存在
   → 直接返回已创建的实例
   → 保持状态（元气值、使用次数等）

3. 用户B (IP: 192.168.1.101) 首次访问
   → user_karma_systems["192.168.1.101"] 不存在
   → 创建新的独立实例
   → 与用户A完全隔离
```

### 2. 请求处理流程

**修改后的代码流程**：
```python
@router.post("/divination/interpret")
async def interpret_divination(request: DivinationRequest, req: Request):
    # 步骤1：从请求中获取用户标识（IP地址）
    client_ip = req.client.host  # 例如："192.168.1.100"
    
    # 步骤2：从字典中获取该用户的元气系统
    # 如果是新用户，defaultdict 会自动创建新实例
    karma_system = user_karma_systems[client_ip]
    
    # 步骤3：使用用户专属的元气系统
    karma_system.update_vitality()  # 只影响当前用户
    preview = karma_system.calculate_cost_preview()  # 当前用户的状态
    karma_system.commit_transaction(cost)  # 只扣除当前用户的元气
    
    # 返回结果
    return response
```

**流程图**：
```
用户请求
    ↓
获取IP地址 (req.client.host)
    ↓
查找字典 user_karma_systems[IP]
    ↓
    ├─ 存在？ → 使用已有实例（保持状态）
    │
    └─ 不存在？ → defaultdict 自动创建新实例（初始状态）
    ↓
使用用户专属的 karma_system
    ↓
处理请求（消耗元气、更新状态等）
    ↓
返回结果
```

### 3. 状态隔离原理

**关键点**：每个用户有独立的实例，状态互不影响

**示例场景**：

```
时间线：
T1: 用户A (IP: 1.1.1.1) 访问
    → user_karma_systems["1.1.1.1"] = KarmaSystem(100%)
    → 算卦，消耗20%
    → 状态：80%

T2: 用户B (IP: 2.2.2.2) 访问
    → user_karma_systems["2.2.2.2"] = KarmaSystem(100%)  ← 新实例！
    → 算卦，消耗20%
    → 状态：80%（但这是用户B的，不影响用户A）

T3: 用户A 再次访问
    → user_karma_systems["1.1.1.1"]  ← 使用已有实例
    → 看到状态：80%（用户A自己的状态）
    → 算卦，消耗20%
    → 状态：60%

T4: 用户B 再次访问
    → user_karma_systems["2.2.2.2"]  ← 使用已有实例
    → 看到状态：80%（用户B自己的状态，不受用户A影响）
```

## 📊 数据结构详解

### 内存布局

**字典结构**：
```python
user_karma_systems = {
    "192.168.1.100": <KarmaSystem对象>,
    "192.168.1.101": <KarmaSystem对象>,
    "192.168.1.102": <KarmaSystem对象>,
    # ...
}
```

**每个 KarmaSystem 对象包含**：
```python
KarmaSystem实例 {
    max_vitality: 100.0,           # 最大元气值
    current_vitality: 80.0,        # 当前元气值
    last_cast_time: 1234567890,    # 上次算卦时间
    last_recovery_time: 1234567890, # 上次恢复时间
    base_cost: 5.0,                # 基础消耗
    penalty_factor: 20.0,          # 惩罚系数
    total_uses: 5,                 # 总使用次数
    last_active_time: 1234567890,  # 上次活跃时间
    # ... 其他属性
}
```

**内存占用估算**：
- 每个 KarmaSystem 实例：约 200-300 字节
- 100 个用户：约 20-30 KB
- 1000 个用户：约 200-300 KB
- 通常可接受

## 🔄 生命周期管理

### 实例创建

**自动创建**：
- 新用户首次访问时，defaultdict 自动创建
- 无需手动管理

**创建时机**：
```python
# 用户首次访问时
karma_system = user_karma_systems["new_ip"]
# ↑ 此时 defaultdict 检测到键不存在
# ↑ 自动调用 lambda: KarmaSystem(...)
# ↑ 创建新实例并存储
```

### 实例保持

**持久化**：
- 实例存储在内存中（字典中）
- 只要服务不重启，状态会一直保持
- 用户再次访问时，使用同一个实例

**状态保持**：
```python
# 用户A第一次访问
karma_system_A = user_karma_systems["1.1.1.1"]
karma_system_A.current_vitality = 80.0

# 用户A第二次访问（可能是几小时后）
karma_system_A = user_karma_systems["1.1.1.1"]  # 同一个实例
print(karma_system_A.current_vitality)  # 仍然是 80.0（如果没恢复）
```

### 实例清理（可选）

**问题**：如果用户很多，内存会一直增长

**解决方案**：定期清理不活跃的用户
```python
import time

def cleanup_inactive_users(max_idle_time=3600):  # 1小时
    """清理1小时未活跃的用户"""
    current_time = time.time()
    inactive_ips = []
    
    for ip, system in user_karma_systems.items():
        idle_time = current_time - system.last_active_time
        if idle_time > max_idle_time:
            inactive_ips.append(ip)
    
    for ip in inactive_ips:
        del user_karma_systems[ip]
    
    return len(inactive_ips)
```

## ⚖️ 优缺点分析

### 优点

1. **简单易实现**
   - 只需修改少量代码
   - 使用标准库，无需额外依赖

2. **自动管理**
   - defaultdict 自动创建实例
   - 无需手动管理生命周期

3. **状态隔离**
   - 每个用户独立状态
   - 互不影响

4. **性能良好**
   - 字典查找 O(1) 时间复杂度
   - 内存占用小

### 缺点

1. **IP 地址限制**
   - 同一网络下的用户共享IP（NAT）
   - 这些用户会共享元气系统

2. **IP 可能变化**
   - 用户切换网络，IP变化
   - 会创建新实例，丢失之前的状态

3. **内存增长**
   - 用户越多，内存占用越大
   - 需要定期清理（可选）

4. **不是真正的用户识别**
   - 基于IP，不是基于用户账号
   - 无法跨设备保持状态

## 🎯 原理总结

### 核心思想

**从"全局单例"改为"字典映射"**：

```
修改前：1个实例 → 所有用户共享
修改后：N个实例 → 每个用户独立
```

### 关键技术

1. **defaultdict**：自动创建默认值
2. **字典映射**：IP → KarmaSystem实例
3. **请求对象**：从 Request 获取用户IP

### 工作流程

```
请求 → 获取IP → 查找/创建实例 → 使用实例 → 返回结果
```

## 💡 进阶方案

如果需要更完善的用户隔离，可以考虑：

1. **基于 Session ID**
   - 前端生成唯一ID
   - 后端使用Session ID作为键

2. **基于 JWT Token**
   - 用户登录后获得Token
   - 从Token中提取用户ID

3. **基于数据库**
   - 用户状态存储在数据库
   - 支持跨设备、跨网络

但对于当前需求，基于IP的方案已经足够！
