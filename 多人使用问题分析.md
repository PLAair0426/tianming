# 多人使用问题分析

## ⚠️ 当前存在的问题

### 1. **全局共享元气系统（严重问题）**

**问题描述**：
- 代码中使用全局单例：`karma_system = KarmaSystem(...)`
- **所有用户共享同一个元气值**
- 用户A消耗元气后，用户B也会受到影响

**影响**：
- 用户A频繁使用，消耗完元气后，所有用户都无法使用
- 用户B可能看到"元气不足"错误，即使他还没用过
- 无法区分不同用户的使用情况

**代码位置**：
```python
# backend/app/routers/divination.py
karma_system = KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0)
```

---

### 2. **Render Free Plan 限制**

**问题描述**：
- 服务在15分钟无活动后会休眠
- 首次请求需要30-60秒冷启动

**影响**：
- 如果服务休眠，所有用户首次访问都需要等待
- 可能被误认为是"用不了"

---

### 3. **没有 API 限流**

**问题描述**：
- 代码中没有实现 API 限流
- 没有并发请求限制

**影响**：
- 如果多人同时使用，可能导致：
  - 资源耗尽（CPU、内存）
  - 响应变慢
  - 服务崩溃
  - DeepSeek API 调用过多，可能触发限流

---

### 4. **没有用户隔离**

**问题描述**：
- 没有用户认证系统
- 没有会话管理
- 无法区分不同用户

**影响**：
- 无法为不同用户提供独立的使用体验
- 无法追踪单个用户的使用情况

---

## ✅ 解决方案

### 方案 1：为每个用户创建独立的元气系统（推荐）

**实现方式**：
- 使用会话ID或IP地址区分用户
- 为每个用户创建独立的 `KarmaSystem` 实例
- 使用字典存储：`{session_id: KarmaSystem}`

**优点**：
- 用户之间互不影响
- 每个用户有独立的元气值
- 实现相对简单

**缺点**：
- 需要会话管理
- 内存占用增加（但通常可接受）

---

### 方案 2：添加 API 限流

**实现方式**：
- 使用 `slowapi` 或 `fastapi-limiter` 库
- 限制每个IP的请求频率
- 例如：每分钟最多10次请求

**优点**：
- 防止滥用
- 保护服务资源
- 防止 DeepSeek API 被限流

**缺点**：
- 需要额外依赖
- 可能影响正常用户（如果限流太严格）

---

### 方案 3：升级 Render 计划

**实现方式**：
- 从 Free Plan 升级到 Starter Plan（$7/月）
- 服务常驻，无冷启动

**优点**：
- 解决休眠问题
- 更好的性能
- 更稳定的服务

**缺点**：
- 需要付费

---

### 方案 4：添加用户认证（长期方案）

**实现方式**：
- 使用 JWT 或 Session 认证
- 为每个用户分配唯一ID
- 在数据库中存储用户状态

**优点**：
- 完整的用户隔离
- 可以追踪使用历史
- 可以添加更多功能（如保存历史）

**缺点**：
- 实现复杂
- 需要数据库
- 需要前端配合

---

## 🚀 快速修复方案（临时）

### 临时方案：基于 IP 地址的用户隔离

可以快速实现基于 IP 地址的用户隔离：

```python
# backend/app/routers/divination.py
from fastapi import Request
from collections import defaultdict

# 存储每个IP的元气系统
user_karma_systems = defaultdict(lambda: KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0))

@router.post("/divination/interpret")
async def interpret_divination(request: DivinationRequest, req: Request):
    # 获取用户IP
    client_ip = req.client.host
    
    # 获取该用户的元气系统
    karma_system = user_karma_systems[client_ip]
    
    # 后续逻辑使用 karma_system
    # ...
```

**优点**：
- 实现简单
- 快速解决多人共享问题
- 不需要数据库

**缺点**：
- 同一网络下的用户会共享（NAT问题）
- IP可能变化
- 不是完美的解决方案

## 📝 代码修改示例

### 修改 divination.py 实现IP隔离

```python
from fastapi import Request
from collections import defaultdict

# 存储每个IP的元气系统
user_karma_systems = defaultdict(
    lambda: KarmaSystem(max_vitality=100.0, base_cost=5.0, penalty_factor=20.0)
)

@router.post("/divination/interpret")
async def interpret_divination(request: DivinationRequest, req: Request):
    # 获取用户IP
    client_ip = req.client.host
    
    # 获取该用户的元气系统
    karma_system = user_karma_systems[client_ip]
    
    # 后续代码使用 karma_system 而不是全局的
    karma_system.update_vitality()
    # ...
```
